# 大模型流式翻译设计

日期：2026-06-19

## 目标

降低大模型翻译的可感知等待时间。卡片不再等待模型生成完整译文后一次性显示，而是在收到首个有效文本分片后立即展示，并持续增量更新。

本次范围包括：

- 默认翻译引擎为“大模型 / OpenAI 兼容接口”时的单引擎翻译。
- 多引擎对照中的大模型结果行。
- 请求取消、超时、兼容回退、缓存和自动化测试。

不包括：

- 更换或自动选择更快的模型。
- 将微软、Google、DeepL 改造成流式接口。
- 句子拆分并发翻译。
- 改变现有提示词的翻译语义。

## 现状与问题

`tcOpenAITranslate` 当前通过 `/chat/completions` 发起非流式请求。后台必须等待完整 JSON 响应，随后才能通过一次性消息把译文交给内容脚本，因此首个可见结果时间等于完整生成时间。

多引擎对照当前使用 `Promise.all`。即使微软或 Google 已完成，界面也要等待所有引擎完成或超时后才统一渲染。大模型既不能增量显示，也会延迟其他引擎结果出现。

现有缓存只存放在 MV3 Service Worker 的内存中。Worker 被浏览器回收后，缓存随之消失。

## 方案选择

采用每张活动卡片一个 `chrome.runtime.Port` 长连接，由后台把各引擎的生命周期事件持续推送给内容脚本。

选择 Port 的原因：

- 适合一个请求对应多个增量事件。
- 长连接可以覆盖完整流式响应生命周期，降低 MV3 Worker 在请求中途休眠的风险。
- Port 断开可作为取消信号，便于中止网络请求。
- 默认大模型翻译和多引擎对照可以共用同一事件协议。
- 项目现有 AI 解释功能已经使用 Port 和 SSE，可复用解析与连接模式。

普通单引擎的微软、Google、DeepL 翻译继续使用现有一次性消息，避免扩大改动范围。

## 组件设计

### 内容脚本：卡片流控制器

每张卡片维护以下状态：

- `_translationPort`：当前翻译连接。
- `_requestSeq`：卡片本地递增序号。
- `_activeRequestId`：当前允许更新界面的请求 ID。
- `_streamBuffers`：按 provider 保存尚未刷新到 DOM 的增量文本。
- `_streamFlushHandle`：批量刷新定时器或动画帧句柄。

卡片发起流式请求时：

1. 取消并断开旧连接。
2. 递增请求序号，生成新的 `requestId`。
3. 建立名为 `tc-translate-stream` 的 Port。
4. 注册消息和断开监听器。
5. 发送请求参数。
6. 只处理与 `_activeRequestId` 相同的事件。

卡片选择新文本、切换语言、切换对照模式或关闭时，必须先取消旧请求。用户主动取消不显示错误。

### 后台：流式翻译会话

后台为每个 `tc-translate-stream` Port 创建一个会话对象：

- 当前 `requestId`。
- 一个会话级 `AbortController`。
- 各 provider 的执行状态。
- 首包和空闲超时计时器。

收到新请求时：

1. 中止该 Port 上仍在执行的旧请求。
2. 验证文本、目标语言和 provider 列表。
3. 读取当前设置。
4. 按 provider 并发启动任务。
5. 每个任务独立发送开始、增量、完成或错误事件。
6. 所有任务结束后发送请求完成事件。

Port 断开时，后台立即调用 `AbortController.abort()` 并清理计时器。

### 大模型 SSE 适配器

新增可复用的大模型流式翻译函数。它负责：

- 使用现有翻译提示词构造请求。
- 请求体设置 `stream: true`。
- 解析 SSE 的 `data:` 行。
- 正确处理一个 JSON 事件跨多个网络数据块的情况。
- 忽略空行、注释和无内容的增量。
- 识别 `data: [DONE]`。
- 聚合完整译文供最终结果和缓存使用。
- 接受 `AbortSignal`、首包回调和增量回调。

AI 解释与翻译可以共用底层 SSE 解析器，但保留各自的提示词和上层事件语义。

## 事件协议

内容脚本发送：

```js
{
  type: 'translate',
  requestId,
  text,
  target,
  context,
  providers
}
```

后台发送：

```js
{ type: 'provider-start', requestId, provider }
{ type: 'provider-delta', requestId, provider, delta }
{ type: 'provider-result', requestId, provider, text, detected, cached, ms }
{ type: 'provider-error', requestId, provider, error, partialText, ms }
{ type: 'request-done', requestId }
```

约束：

- `provider-delta` 仅用于支持流式输出的 provider。
- `provider-result` 是该 provider 的权威最终状态。
- 缓存命中直接发送 `provider-result`，不发送虚假的逐字增量。
- `provider-error.partialText` 只在流中途失败且已有内容时提供。
- 所有事件都必须携带 `requestId`，内容脚本忽略过期事件。

## 单引擎数据流

当默认 provider 不是大模型时，保留现有 `tc-translate` 消息流程。

当默认 provider 是大模型时：

1. 卡片先显示“翻译中…”。
2. 建立流式 Port 并发送只包含 `openai` 的 provider 列表。
3. 收到首个 `provider-delta` 时移除加载占位，立即显示内容。
4. 后续增量进入缓冲区，按批次刷新。
5. 收到 `provider-result` 后以最终文本校准界面并更新复制、朗读、收藏所需状态。
6. 收到错误时，按是否已有部分内容选择完整错误或中断提示。

## 多引擎对照数据流

开启对照模式后，不再等待 `Promise.all` 完成再构造整个结果列表。

1. 内容脚本立即按设置中的 provider 顺序创建全部结果行。
2. 每行初始状态为“等待中”。
3. 后台并发启动所有 provider。
4. 微软、Google、DeepL 完成后，各自立即通过 `provider-result` 更新对应行。
5. 大模型收到首个分片后立即更新对应行，并持续增量显示。
6. 单个 provider 失败只更新自己的行，不影响其他 provider。
7. `request-done` 只代表该批次已全部结束，不触发结果重排。

卡片顶部的复制、朗读和收藏继续使用第一个成功 provider 的最终结果。部分流式文本不作为主结果，避免用户保存未完成译文。

## 渲染节流

网络分片可能非常细。内容脚本不在每个 `delta` 上直接修改 DOM，而是：

1. 将增量追加到 provider 对应的内存缓冲区。
2. 若没有待执行刷新，则安排一次约 40ms 后的刷新。
3. 一次刷新合并期间收到的全部增量。
4. 完成、错误、取消或关闭卡片时清理刷新句柄。

这会把可见更新频率控制在约每秒 25 次，同时仍保持连续输出的观感。

## 取消与竞态控制

以下操作必须取消当前翻译：

- 卡片被复用于新的选中文字。
- 目标语言改变。
- 单引擎和对照模式切换。
- 卡片关闭。
- 卡片发起新的重试。

取消分为两层：

- 内容脚本断开旧 Port，阻止继续接收事件。
- 后台监听断开并中止对应 fetch，释放网络和计时器。

即使取消与消息到达发生竞态，`requestId` 检查仍保证旧请求不能覆盖新内容。

## 超时

大模型流式请求采用两阶段超时：

- 首包超时：请求发出后 12 秒仍未收到有效文本分片，则中止并报错。
- 流空闲超时：已经开始输出后，连续 20 秒没有新的有效文本分片，则中止并标记输出中断。

收到每个有效文本分片时重置空闲超时。普通非流式 provider 继续使用现有超时策略。

## 兼容回退

部分 OpenAI 兼容服务可能忽略 `stream: true`，或以普通 JSON 返回结果。

适配器根据响应头和首个数据块判断响应形式：

- `text/event-stream`：按 SSE 解析。
- JSON 响应：按现有非流式结构解析，并通过一次 `provider-result` 返回。
- 无法识别的成功响应：尝试解析 JSON；失败后返回明确的协议错误。

回退仅保证功能可用，不伪造流式效果。

## 错误处理

- HTTP 非成功状态：解析兼容接口常见的嵌套或顶层错误消息。
- 首包超时：显示“大模型响应超时，可重试”。
- 流中断且没有文本：显示翻译失败。
- 流中断且已有文本：保留部分译文，并显示“输出中断，可重试”。
- 用户主动取消：不显示错误。
- 单个对照引擎失败：仅该行进入失败状态。
- Port 异常断开：当前请求仍有效时才显示连接中断。

任何不完整结果都不得写入缓存。

## 缓存设计

保留现有内存 LRU，并增加 `chrome.storage.local` 持久缓存层。

缓存键继续由以下内容组成：

- provider。
- 目标语言。
- 原文。
- 大模型上下文哈希。
- 大模型接口地址和模型标识的哈希。
- 提示词协议版本。

加入接口、模型和提示词版本可以防止用户切换配置后命中语义不一致的旧结果。API Key 不进入缓存键或缓存内容。

读取顺序：

1. 内存 LRU。
2. 持久缓存。
3. 网络请求。

持久缓存要求：

- 仅保存完整成功结果。
- 最多保留 500 条。
- 每条记录保存最后访问时间。
- 默认存活 30 天。
- 写入和清理使用低频批处理，避免每个 token 写 storage。
- 命中持久缓存后回填内存 LRU。

持久缓存属于本次交付范围。实施时应先稳定流式、取消和竞态控制，再接入持久缓存，避免缓存逻辑干扰流式主路径的调试。

## 测试计划

### SSE 解析

- 一个事件完整落在一个数据块。
- 一个 JSON 事件横跨多个数据块。
- 一个数据块包含多个事件。
- 混合 CRLF 和 LF。
- keep-alive、空行和无内容 delta。
- 正确处理 `[DONE]`。
- 流结束时仍有尾部缓冲内容。

### 单引擎翻译

- 首个 delta 到达后立即替换加载状态。
- 最终结果校准流式文本并成为可复制、收藏的结果。
- JSON 回退能显示完整译文。
- 完整成功结果写入缓存。
- 中断的部分结果不写缓存。

### 多引擎对照

- 所有行立即出现。
- 普通引擎按完成顺序独立更新。
- 大模型行增量更新，不阻塞其他行。
- 一个引擎失败不影响其他行。
- 请求全部结束后状态正确。

### 取消与竞态

- 新选择取消旧请求。
- 切换目标语言取消旧请求。
- 切换对照模式取消旧请求。
- 关闭卡片取消旧请求。
- 过期 `requestId` 的事件无法修改当前卡片。
- 用户取消不显示错误。

### 缓存

- 内存命中不发网络请求。
- 持久缓存命中回填内存。
- 不同模型、接口和提示词版本不会互相命中。
- 半截结果不缓存。
- 容量与过期清理符合限制。

## 验收标准

- 默认大模型翻译在首个文本分片到达时立即显示，不等待完整响应。
- 多引擎对照中的每个 provider 独立显示进度和结果；大模型行可以流式更新。
- 任何旧请求都不能覆盖新选择、新语言或新模式的内容。
- 关闭卡片或切换请求后，后台网络请求会被中止。
- 不支持 SSE 的兼容接口仍可通过普通 JSON 返回结果。
- 只有完整成功译文进入缓存。
- 现有微软、Google、DeepL 单引擎翻译行为保持不变。
- 自动化测试覆盖流解析、事件调度、取消竞态、回退和缓存边界。

2. 实现大模型流式翻译适配器及 JSON 回退。

7. 接入持久缓存并验证容量、过期和配置隔离。
